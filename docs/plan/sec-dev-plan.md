# 二次开发计划

## 一、功能扩展

### 1. 自定义回复模式

- 系统自带回复模式：抑郁预测模式、恋爱模式、高情商模式
- 用户可以自定义设置回复模式，当前想法是通过提示词来进行规范回复的方向以达到“特定回复模式”的效果。
- 自定义模式切换：暂定两种方式，“主界面下拉框选择”，“主界面新增左侧菜单用以切换多个模式子界面”

### 2. 自动回复

- 将自动回复抽离出来，用户通过勾选“自动回复”设置项来开启自动回复功能。
- 新增一个“推荐回复”文本框，框内主要展示系统根据用户聊天内容给出的回复。文本框增加“一键粘贴”按键，用户点击之后可以将推荐回复直接粘贴到聊天窗口。
- 新增一个“一键回复”按键，虽然用户没有勾选自动回复设置项，但是若是“推荐回复”文本框有内容，那么就可以点击该按键来执行自动回复的操作。
- 自动回复的情况有以下几种：设置界面中的“自动回复”针对所有“非特定对象”，在添加特定对象时，该弹窗中也有“自动回复”，专门针对该特定对象是否开启自动回复。

### 3. 特定模式针对特定对象

- 用户可以通过聊天软件的名字来设置特定对象，系统在检测到红点之后，会获取该红点对应的对象的名字，来判断是否为当前模式的特定对象，若是，则会根据当前模式的回复规则来回复。若不是，则跳过。
- 特定对象的属性有：名称（必填）、特定称呼（非必填）、关系（非必填）、模式（默认是当前子界面的模式）、自动回复。
- 无特定对象，特定对象视图为空。
- 所有模式设置的特定对象作为一个集合（只有启动了模式的特定对象才会被加入集合），系统监测到红点并提取到名称之后，在整个集合中搜索，若是搜到，则根据mode数据分发到对应的模式进程进行回复。若是搜不到，则使用设置中设置的全局默认模式进行回复。

### 4. 多线程处理

- 针对本系统新增的多对话模式，每一个模式都可以启动作为一个子进程去进行回复，每个模式间并行，互不干扰。
- 同时要注意，在每个模式去进行操作、回复时，要确保这个操作是只能被一个线程来执行，否则会导致线程安全问题。

### 5. 供应商接口拓展

- 现如今项目只支持火山方舟 (Volcengine Ark) 供应商的模型，后续可以考虑支持其他供应商的模型。
- 针对对应的供应商，支持用户配置指定的模型，用户可以在设置界面中选择不同的模型作为全局默认模型。

## 二、界面扩展

### 1.主界面

#### 1.1 自定义回复模式菜单

- 主界面左侧新增一个“自定义回复模式”菜单，其中罗列有系统自带、用户自定义的回复模式，用户可以通过点击菜单来切换不同的回复模式。
- 菜单底部有一个“添加自定义模式”按键，用户点击之后可以打开一个弹窗，用于添加新的自定义回复模式。
- 设置、工作记忆按键转移到此处菜单的最底部，与上方模式部分用一个横线组件分隔开。

#### 1.2 自定义回复模式弹窗

- 模式名称输入框：用户输入自定义的回复名称，例如“抑郁预测”、“恋爱”、“高情商”等。
- 回复规则输入框：用户输入自定义的回复规则，主要是当前模式下的ai回复的引导提示词。

#### 1.3 模式子界面

- 将当前主界面的主要组件封装成一个固定的模式子界面，每新增一个模式，就新增一个对应的模式子界面，每个模式子界面之间是独立的，互不干扰的。
- 模式名称：展示当前模式的名称。
- 运行状态：例如原界面中的待定。
- 目标应用：展示当前模式的目标应用，例如“微信”、“QQ”等。
- 特定对象视图：展示当前模式的特定对象，用户可以在视图中添加、删除特定对象。
- 运行日志视图：展示当前模式的运行日志，用户可以查看系统在当前模式下的运行情况。
- 自动回复选项：用户可以通过勾选“自动回复”设置项来开启自动回复功能。
- 启动/停止按键：用户点击该按键来启动或停止当前模式的子进程。

### 2. 设置界面

#### 2.1 整体布局

- 基础配置：用户可以在这里设置全局的默认模型、默认模式等。
- 模式管理：用户可以在“模式管理”中统一管理系统自带和自定义模式。
- 模型配置：模型配置部分，用户可以在这里自定义添加不同的模型。
- 模型训练：将情感模型训练移到“模型训练”部分。
- 智能体：按照原项目设置界面的智能体不变。

#### 2.2 基础配置

- 全局默认模型，用户可以通过下拉框选择已经配置好的模型作为全局默认模型，这里的全局指的是系统流程中使用到的步骤，例如截图分析、大模型回复等。
- 全局默认模式，用户可以通过下拉框选择已经自定义好的模式作为全局默认模式，这里的全局指的是系统流程中识别到红点后打开对话框，在获取到对象名称之后，进行任务分发。例如若是当前对象不属于任何一个模式的特定对象，那么就分配到全局默认模式去进行分析、回复。
- 自动回复选项：用户可以通过勾选“自动回复”设置项来开启自动回复功能。

#### 2.3 模式管理

- 模式展示视图：可以在视图中查看已经自定义的模式。点击某个模式之后，弹窗显示模式的具体信息。在视图顶部有“选择模式”的下拉框，主要是控制视图中展示的已设置模式。
- 每个模式名称后都有一个标签，暂定有两种标签：一种是模式的来源“系统/自定义”（可以使用蓝色的字来特别标识），系统自带的模式不提供删除选项；另一种是模式的运行状态，包括“已启动”、“未启动”（可以使用绿色和红色的字来分别标识），用户可以选择其运行状态来控制模式菜单的展示情况（类似于实例，若是未启用，则不需要实例化对应模式子界面的组件，减少组件带来的开销）。
- 每个模式右侧都有一个删除按键，用户点击之后可以删除该模式。

#### 2.4 模型配置

- 添加模型按键：用户点击之后弹出窗口，其中包含了设置模型的一些组件“供应商下拉框”、“模型名称输入框”、“API Key输入框”、“base URL输入框”以及确定或者取消按键。（注意在添加时，某些必填项若为空，需要提示用户）
- 模型展示视图：可以在视图中查看已经添加的模型。点击某个模型之后，弹窗显示模型的具体信息。在视图顶部有“选择供应商”的下拉框，主要是控制视图中展示的已设置模型。
- 每个模型右侧都有一个删除按键，用户点击之后可以删除该模型。

#### 2.5 模型训练

- 将原项目中的情感模型训练移到此处，用户可以通过开始训练和停止训练来进行情感模型的训练，可以通过训练日志视图来查看训练进度。
- 训练日志视图：展示情感模型的训练进度，包括训练轮数、验证集最优模型保存路径等。
- 增加python环境变量测试，用户可以在设置界面中测试python环境变量是否配置正确，若未配置正确，则会提示用户配置python环境变量。
- 增加python依赖安装，用户可以在设置界面中安装python依赖，若未安装，则会提示用户安装python依赖。

#### 2.6 智能体

- 智能体：按照原项目设置界面的智能体不变。

#### 3.1 性能优化

#### 3.2 提示词优化

- 针对VLM的工作提示词可以进行优化，现如今只是针对对方发送过来的最后一句话进行识别提取，后续可以提取窗口内，双方的所有对话内容，作为生成回复的依据。
- 针对模型通过截图内容进行的回复，后续可以根据双方的具体聊天内容进行着重回复，例如说：若是对方在前面发送了信息，同时我方已经进行回复了，那么可以针对后续对方发来的内容进行前后情感变化分析（即对方在查看我方回复后，语言上情绪的变化），再结合着前后语境进行回复。

#### 3.3 检测、运行优化

- 对于现如今的流程，VLM会隔段时间一直检测，无论对话窗口是否变化。现在添加了半自动回复逻辑，用户可以取消自动回复，因此，若是系统给出推荐回复，但是用户一直没有进行操作时，系统会一直检测同一个界面，无论当前对话窗口是否有发生变化（例如对方持续发送信息），系统都会当作是一个全新的界面来检测，并重新给出“推荐回复”。
- 若是用户一直没有操作，又没有设置自动回复，那么界面就会一直没有变化，系统就会对着同一个（可能没有变化）对话窗口持续检测并给出回复，一直循环，即便有新的消息也无法操作界面打开。但实际上这是我们不希望的。
- 我们希望：系统针对当前对话给出推荐回复之后，能够提示用户（可以在左侧菜单栏的当前模式项的右侧使用一个红色感叹号标识），若是用户一直没有操作（VLM检测对话界面一直没有发生变化），则进入待机状态（遵循少做少错的标准）。所以这里可以改一下状态，绿色表示运行中，蓝色表示启动中，红色表示待机中，灰色表示未启动。
- 这里需要说明一下，本系统遵循“少做少错”的标准，但是实际上本系统最大的两个应用场景应该是：完全自动回复（智能客服）；需要系统的模型输出来辅助聊天（恋爱模式）。而并不适合设置了未自动回复之后长时间不关注。因此这里的优化主要是针对各种可能出现的情况，来设计一个应急措施，所以遵循“少错少错”。
- 具体实现设想：可以将上一次的检测截图保存，在新一次的检测中，将新旧两个截图一起发送给视觉模型，由模型进行分析，若是两张截图的对话相关内容基本不变（这里指的是对话窗口里面的对话内容基本不变，实际上可能会有一些细微变化，但是只要不是对方发送了新的消息，则认定为没有变化），则返回“待机”，系统收到这样的回复之后就进入待机状态。因为在视觉模型返回之后才通过对象名称去路由，然后才会进入模式部分，因此，只需要在收到“待机”之后跳过后续所有操作，就可以实现“待机”的效果了。后续VLM会持续将新旧两张截图再发送给视觉模型，持续循环。
- VLM返回内容设计：首先要弄清楚现如今的实现中，VLM分析截图之后返回的内容都有哪些属性，猜测应该有“对象名称”、“模型的推荐回复”。同时要弄清回复的内容是通过什么进行控制的，若是提示词，则修改一下提示词，让返回结果分两种情况，即是否有变化，有变化则正常返回，没有变化则名称正常返回、推荐回复返回空，系统检测到名称为特定字段“待机”，且推荐回复为空，则不更新推荐回复，同时进入待机模式。
- 现在有个新问题，系统的流程是怎样的？现在看来，是检测是否有未读信息，而不是一直截图查看对话界面。这样的逻辑下需要重新思考一下上述实现。
