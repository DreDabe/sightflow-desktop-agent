# 拓展计划

本文件记录系统完成第一个版本开发后的拓展计划，暂时不需要理会。

## 拓展1：模式级待机与消息队列

### 问题场景

模式 A 进入待机 → 系统去处理模式 B 的未读消息 → 此时模式 A 的另一个特定对象发来新消息 → 系统路由分发到模式 A → 但模式 A 正在待机

### 用户设想

为每个模式增加一个消息队列，保存待处理的联系人名称和回复状态，模式通过队列逐步处理。

### 分析与改进

1. **队列项应包含**：
   - 联系人名称
   - 路由到的模式ID
   - 入队时间
   - 状态（pending/processing/done）

2. **不需要保存截图/文本**：因为处理队列时需要重新截图获取最新内容，保存旧截图没有意义

3. **队列处理时机**：
   - 模式退出待机后，检查队列是否有待处理项
   - 若有，按入队时间顺序逐个处理
   - 处理方式：搜索联系人 → 打开对话 → 截图分析 → 生成回复

4. **与现有架构的兼容性**：
   - 当前 `GenericChannelSession` 是单会话的
   - 需要引入**会话调度器**（SessionScheduler），管理多个模式的会话切换
   - 调度器负责：决定当前处理哪个模式的对话、切换对话窗口、管理队列

### 模式级待机的完整流程

```
检测到未读消息
  │
  ├─ 识别对象名称
  ├─ 路由到模式
  │
  ├─ IF 目标模式未待机:
  │   └─ 正常处理（打开对话 → 分析 → 回复）
  │
  └─ IF 目标模式已待机:
      ├─ 加入该模式的待处理队列
      └─ 不打断当前正在处理的模式

模式退出待机时:
  │
  ├─ 检查待处理队列
  ├─ IF 队列非空:
  │   ├─ 取出队首项
  │   ├─ 搜索联系人 → 打开对话
  │   ├─ 截图分析 → 生成回复
  │   └─ 处理完后检查下一项
  │
  └─ IF 队列为空:
      └─ 恢复正常检测循环
```

### 需要新增的功能

1. **搜索联系人打开对话**：通过微信搜索框输入联系人名称，打开特定对话
2. **会话调度器**：管理多模式会话切换，避免窗口操作冲突
3. **模式消息队列**：每个模式维护自己的待处理消息队列


## 拓展2：提示词模块

### 重构设想

1. 将原本系统中的提示词抽象封装成一个模块，与模型调用模块分离。在目录结构中创建一个 `prompt` 目录，用于存放所有提示词文件。

```
prompt
  │
  ├─ system
  └─ mode
```

2. 自定义模式部分涉及到引导模型进行回复的规则提示词，每个模式都与回复的规则提示词相对应。因此，若是新增一个模式，就相当于实例化一个提示词模块。
3. 系统提示词属于开发者配置，对用户透明，用户无需直接配置、修改。
4. 动态拼接功能：因为在本系统中拥有“特定对象”的概念，其中特定对象中包含的属性包括联系人的名称、关系、称呼，因此，若是设置有这些属性，需要模型从这样特定的角度去进行回复，因此需要将特定对象的属性动态拼接到提示词中，在模型调用时传递给模型。另外，若是后续实现了特定对象的记忆系统，也需要将记忆信息也动态拼接到提示词中。


## 拓展3：特定对象记忆系统

### 问题场景

本系统最大的一个特色为针对特定对象应用特定的回复模式，但是，目前系统只能针对当前对话窗口中实际存在的消息文本进行回复，这种回复没有前后文语境信息，因此给出的回复质量较低。若是能够将此前的对话记录也考虑进去，那么回复的质量会提高。

### 实现设想

1. 每个特定对象新增一个“对话记忆”属性，用来记录该特定对象的对话记录。
2. 对话记录的逻辑为：每次系统收到特定对象的消息时，都会进行系统流程，但是实际上，系统出现会出现很多情况（例如：系统待机状态、对方持续发送消息、用户手动操作发送消息），这样就会导致一种情况，若是仅仅只是使用脚本去追加对话记录，会出现缺失、重复的情况，这样会影响对话记忆的准确性。
3. 因此，这里推荐，记忆记录的功能可以通过文本模式全程参与，这样能够很好的整理对话记录，唯一的问题是增加了调用API的开销。
4. 这里的记忆记录不仅仅只是记录对方的消息，也要记录己方的消息，这样才能够完整地记录对话记录，为后续的回复提供上下文信息和语境。


## 拓展4：用户回复文本

### 问题场景

因为用户可能想要自己输入文本，然后希望系统使用用户的输入作为文本基础、模式的回复规则作为回复的修改方向，也就是用户提供原始文本，系统根据回复规则对原始文本进行修改，生成最终的回复。

### 实现设想

1. 添加一个“用户回复”的文本输入框。框内添加“清除”按键，用户点击后可以清除空文本框。
2. 若是文本框为空时，默认使用系统回复文本，若是文本框不为空，则将用户输入的原始文本动态拼接到提示词中，在模型调用时传递给模型。
3. 系统根据提示词，对原文本按照回复规则进行修改、粉饰，生成最终的回复。
