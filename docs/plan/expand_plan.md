# 3.3 检测、运行优化 — 详细方案

## 一、问题定义

### 1.1 当前问题

在自动回复关闭（半自动回复模式）的情况下，系统存在以下问题：

1. **重复分析**：像素 diff 对任何像素变化都敏感（时间戳更新、输入状态提示、滚动条变化等），导致系统对实质上相同的对话内容反复执行完整的 VLM 分析 + 回复生成流程
2. **推荐回复覆盖**：系统反复生成推荐回复，覆盖之前的内容，用户来不及操作
3. **API 成本浪费**：VLM 和回复模型的 API 调用持续消耗，即使对话内容无实质变化
4. **无法暂停**：系统没有"暂停等待"的机制，无法在给出推荐回复后等待用户操作

### 1.2 优化目标

1. 系统给出推荐回复后，若对话内容无实质变化且用户未操作，进入**待机状态**
2. 待机状态下不执行 VLM 分析、Provider 调用、红点检测等高成本操作
3. 待机状态下只检测"用户是否操作"和"当前对话是否有新消息"
4. 用户操作或对话变化时，退出待机恢复正常运行

---

## 二、待机状态设计

### 2.1 待机级别

#### 第一阶段：系统级待机（当前实现）

- 待机状态是系统级的，所有模式都进入待机
- 待机期间不处理任何新未读消息
- 只等待当前对话变化或用户操作

#### 第二阶段：模式级待机（后续升级）

- 每个模式独立管理自己的待机状态
- 模式 A 待机时，模式 B 仍可正常处理其特定对象的新消息
- 需要新增"搜索联系人打开对话"功能（用于用户操作后重新打开待机模式的对话窗口）
- 需要引入模式级消息队列（见第四节）

### 2.2 待机触发条件

- 系统给出推荐回复后，**连续 N 轮**（建议 N=2）检测到对话无实质变化，才进入待机
- 避免因单次像素噪声误判而进入待机

### 2.3 待机覆盖范围

待机状态下，以下操作**暂停**：

- `observe_chat`：不重新分析对话、不调用 Provider
- `check_unread` 中的红点/未读区域检测：不模拟点击打开新对话

待机状态下，以下操作**继续**：

- 像素 diff：检测当前已打开对话窗口是否有变化
- 用户操作检测：检测用户是否点击了一键跳过/一键粘贴/一键回复

### 2.4 退出待机的条件

| 条件 | 说明 |
|---|---|
| 对话内容发生实质变化 | 像素 diff → 文本提取对比确认有新消息 |
| 用户点击"一键跳过" | 跳过本次推荐回复，退出待机 |
| 用户点击"一键粘贴" | 粘贴推荐回复到剪贴板，退出待机 |
| 用户点击"一键回复" | 发送推荐回复，退出待机 |

### 2.5 待机期间的检测频率

采用**渐进式退避**策略：

| 轮次 | 检测间隔 |
|---|---|
| 第1轮 | 5秒 |
| 第2轮 | 10秒 |
| 第3轮 | 20秒 |
| 第4轮+ | 60秒（上限） |

用户操作后，重置为5秒。

### 2.6 自动回复与待机

- **全模式全对象自动回复**：无待机，系统自动发送回复，不会出现"用户不操作"的情况
- **存在任一模式/对象未开启自动回复**：仍需待机机制，因为可能出现推荐回复后用户未操作的情况

## 三、语义确认方案

### 3.1 方案选择：独立文本提取 + 对比

像素 diff 检测到变化后，调用 `extractChatText` 提取当前文本，与上次保存的文本对比，确认有实质变化才退出待机进入 `observe_chat`。

### 3.2 实现细节

```
check_unread 阶段:
  │
  ├─ 像素 diff (hasChatAreaChanged)
  │   ├─ 无变化 → wait_retry（渐进式退避）
  │   └─ 有变化 ↓
  │
  ├─ [待机状态专属] 文本提取 + 对比
  │   ├─ extractChatText(screenshot) → currentText
  │   ├─ 对比 currentText 与 lastExtractedText
  │   ├─ 无实质差异 → 继续待机，wait_retry
  │   └─ 有实质差异 → 退出待机 → observe_chat
  │
  └─ [非待机状态] 正常流程
      ├─ hasDiff → observe_chat
      └─ 无 diff → 检测红点/未读
```

### 3.3 需要新增的状态

```typescript
interface GenericChannelState {
  // ... 现有字段
  standby: boolean                    // 是否处于待机状态
  standbySince: number | null         // 待机开始时间
  standbyRetryCount: number           // 待机期间的重试次数（用于渐进式退避）
  lastExtractedText: string | null    // 上一次提取的聊天文本（用于对比）
  lastRecommendReplyModeId: string | null  // 上一次推荐回复对应的模式ID
}
```

### 3.4 文本对比策略

- 简单字符串对比：`currentText.trim() !== lastExtractedText.trim()`
- 若需要更精确的对比，可使用 diff 算法计算差异比例，超过阈值（如10%）才认为有实质变化
- 初期建议使用简单字符串对比，后续根据实际效果优化

## 四、模式级待机与消息队列（第二阶段）

### 4.1 问题场景

模式 A 进入待机 → 系统去处理模式 B 的未读消息 → 此时模式 A 的另一个特定对象发来新消息 → 系统路由分发到模式 A → 但模式 A 正在待机

### 4.2 用户设想

为每个模式增加一个消息队列，保存待处理的联系人名称和回复状态，模式通过队列逐步处理。

### 4.3 分析与改进

用户设想的方向正确，但需要补充以下内容：

1. **队列项应包含**：
   - 联系人名称
   - 路由到的模式ID
   - 入队时间
   - 状态（pending/processing/done）

2. **不需要保存截图/文本**：因为处理队列时需要重新截图获取最新内容，保存旧截图没有意义

3. **队列处理时机**：
   - 模式退出待机后，检查队列是否有待处理项
   - 若有，按入队时间顺序逐个处理
   - 处理方式：搜索联系人 → 打开对话 → 截图分析 → 生成回复

4. **与现有架构的兼容性**：
   - 当前 `GenericChannelSession` 是单会话的
   - 需要引入**会话调度器**（SessionScheduler），管理多个模式的会话切换
   - 调度器负责：决定当前处理哪个模式的对话、切换对话窗口、管理队列

### 4.4 模式级待机的完整流程（第二阶段）

```
检测到未读消息
  │
  ├─ 识别对象名称
  ├─ 路由到模式
  │
  ├─ IF 目标模式未待机:
  │   └─ 正常处理（打开对话 → 分析 → 回复）
  │
  └─ IF 目标模式已待机:
      ├─ 加入该模式的待处理队列
      └─ 不打断当前正在处理的模式

模式退出待机时:
  │
  ├─ 检查待处理队列
  ├─ IF 队列非空:
  │   ├─ 取出队首项
  │   ├─ 搜索联系人 → 打开对话
  │   ├─ 截图分析 → 生成回复
  │   └─ 处理完后检查下一项
  │
  └─ IF 队列为空:
      └─ 恢复正常检测循环
```

### 4.5 需要新增的功能

1. **搜索联系人打开对话**：通过微信搜索框输入联系人名称，打开特定对话
2. **会话调度器**：管理多模式会话切换，避免窗口操作冲突
3. **模式消息队列**：每个模式维护自己的待处理消息队列

## 五、UI 变更

### 5.1 新增"一键跳过"按钮

- 位置：推荐回复文本框下方，与"一键粘贴"、"一键回复"同行
- 功能：跳过本次推荐回复，退出待机状态
- 行为：
  - 保留推荐回复文本框内容（不清空）
  - 退出待机状态
  - 系统给出提示（UI文字提示 + 日志输出 + 状态标识变化）
  - 恢复正常检测频率（5秒）

### 5.2 待机状态标识

- 左侧菜单栏当前模式项右侧：红色圆点/感叹号
- 推荐回复文本框下方：显示"待机中"文字提示
- 运行日志：输出"已进入待机状态"日志

### 5.3 模式状态标识更新

| 状态 | 颜色 | 说明 |
|---|---|---|
| 未启动 | 灰色 | 模式未启动 |
| 启动中 | 蓝色 | 模式正在启动 |
| 运行中 | 绿色 | 模式正常运行 |
| 待机中 | 红色 | 模式进入待机 |
