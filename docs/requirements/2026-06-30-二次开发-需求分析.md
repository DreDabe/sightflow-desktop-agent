## 1. 项目背景

### 1.1 项目概述

SightFlow Desktop Agent 是一个基于 Electron 的桌面端 RPA (Robotic Process Automation) 自动回复系统，核心功能是监控即时通讯软件的聊天窗口，通过 AI 视觉分析理解对话内容，自动生成并发送回复。

本次二次开发旨在将原有的单一回复模式扩展为**多模式并行运行**架构，支持用户自定义回复风格、针对特定对象路由、半自动回复等能力，使系统从"单一自动回复工具"升级为"多模式智能回复平台"。

### 1.2 目标用户

- 需要同时维护多种社交关系的用户（如同时处理工作、家庭、朋友等不同社交场景）
- 希望针对不同对话对象使用不同回复风格的用户
- 需要在自动回复和手动回复之间灵活切换的用户

### 1.3 项目目标

1. 支持用户自定义多种回复模式，每种模式有独立的 prompt、情感分析配置、特定对象列表
2. 支持特定对象路由：系统识别对话对象后，自动分发到对应模式处理
3. 支持半自动回复：用户可选择自动回复或推荐回复（手动确认后发送）
4. 支持多供应商模型配置，用户可灵活切换 AI 模型
5. 保持与现有架构的兼容性，最小化对核心流程的侵入性修改

### 2.2 功能详述

#### F001 自定义回复模式

- **触发条件：** 用户在左侧菜单点击"添加自定义模式"或在设置界面模式管理中添加
- **前置条件：** 无
- **主要流程：**
  1. 用户点击"添加自定义模式"按键
  2. 弹出"添加自定义模式"弹窗
  3. 用户填写模式配置项
  4. 点击确定，创建模式并持久化
  5. 左侧菜单新增该模式条目，右侧新增该模式子界面组件实例
- **异常处理：**
  - 模式名称为空时，提示用户"模式名称不能为空"
  - 回复规则为空时，提示用户"回复规则不能为空"
  - 模式名称重复时，提示用户"模式名称已存在"
- **输出结果：** 新增一个自定义回复模式，出现在左侧菜单和设置界面的模式管理中

**模式数据结构：**

```typescript
interface ReplyMode {
  id: string
  name: string
  source: 'system' | 'custom'
  prompt: string
  sentimentEnabled: boolean
  unifiedPrefix: string
  enabled: boolean
  running: boolean
  specificObjects: SpecificObject[]
  autoReply: boolean
  createdAt: number
  updatedAt: number
}
```

**模式配置项（添加弹窗字段）：**

| 字段            | 类型      | 必填 | 说明                      |
| ------------- | ------- | -- | ----------------------- |
| 模式名称          | string  | 是  | 如"抑郁预测"、"恋爱"、"高情商"      |
| 回复规则 (Prompt) | string  | 是  | AI 回复的引导提示词             |
| 情感分析开关        | boolean | 否  | 默认关闭，开启后该模式运行时使用情感分析    |
| 统一开头          | string  | 否  | 如"【机器客服自动回复】"，回复内容前自动添加 |

**系统预设模式：**

| 模式名称 | Prompt 方向   | 情感分析 | 可删除 | 可禁用 |
| ---- | ----------- | ---- | --- | --- |
| 抑郁预测 | 抑郁倾向识别与关怀回复 | 开启   | 否   | 是   |
| 恋爱   | 恋爱风格回复      | 关闭   | 否   | 是   |
| 高情商  | 高情商社交回复     | 关闭   | 否   | 是   |

#### F002 特定对象路由

- **触发条件：** 系统检测到红点消息
- **前置条件：** 至少有一个模式处于"运行"状态
- **主要流程：**
  1. 全局调度器检测到红点
  2. 点击红点进入对话窗口
  3. VLM 识别当前对话对象的名称（从标题栏提取）
  4. 在所有"运行中"模式的特定对象集合中搜索匹配
  5. 匹配成功 → 路由到对应模式进行回复
  6. 匹配失败 → 路由到全局默认模式进行回复
  7. 对应模式生成回复后，根据自动回复设置决定发送方式
- **异常处理：**
  - VLM 识别失败 → 使用全局默认模式处理
  - 全局默认模式在"模式管理"中被设置为"未启用"，或该模式未启动运行 → 跳过本次回复，记录日志
  - 添加特定对象时，对象名称为空 → 提示用户"对象名称不能为空"
  - 添加特定对象时，对象名称与已存在的特定对象集合中任一名称完全相同 → 拒绝添加，提示用户"该对象名称已存在于[模式名]模式中"
- **输出结果：** 消息被正确路由到对应模式处理

**特定对象数据结构：**

```typescript
interface SpecificObject {
  id: string
  name: string
  title: string
  relationship: string
  modeId: string
  autoReply: boolean | null
}
```

**特定对象配置项（添加弹窗字段）：**

| 字段   | 类型       | 必填 | 说明                    |
| ---- | -------- | -- | --------------------- |
| 名称   | string   | 是  | 对方在聊天软件中的名称，用于 VLM 匹配 |
| 特定称呼 | string   | 否  | 我方对对方的称呼，注入 prompt    |
| 关系   | string   | 否  | 与对方的关系，注入 prompt      |
| 模式   | dropdown | 否  | 默认为当前模式，可改为其他已启用模式    |
| 自动回复 | boolean  | 否  | 默认跟随模式全局设置，可单独覆盖      |

**路由分发流程图：**

```
红点检测 → 点击进入对话 → VLM识别对象名称
                                    ↓
                        ┌───────────────────────┐
                        │ 在特定对象集合中搜索    │
                        └───────┬───────┬───────┘
                                │       │
                          匹配成功   匹配失败
                                │       │
                                ↓       ↓
                        对应模式处理  全局默认模式处理
                                │       │
                                ↓       ↓
                        ┌───────────────────────┐
                        │ 根据自动回复设置决定    │
                        │ 自动发送/推荐回复/跳过  │
                        └───────────────────────┘
```

#### F003 半自动回复

- **触发条件：** 模式生成回复后
- **前置条件：** 模式处于"运行"状态
- **主要流程：**
  1. 模式通过 Provider 生成回复文本
  2. 回复文本**始终**显示在"推荐回复"文本框中
  3. 根据自动回复设置决定后续行为：
     - **自动回复开启** → 在推荐回复展示的同时，自动粘贴到聊天输入框 + 模拟 Enter 发送
     - **自动回复关闭** → 无后续操作，等待用户手动操作
  4. 推荐回复文本框中的内容，用户可：
     - 点击"一键粘贴" → 自动粘贴到聊天输入框，不发送（用户可修改后手动发送）
     - 点击"一键回复" → 自动粘贴到聊天输入框 + 模拟 Enter 发送
- **异常处理：**
  - 聊天窗口未激活时，粘贴操作失败 → 提示用户"请先激活聊天窗口"
  - 推荐回复为空时，"一键粘贴"和"一键回复"按键置灰不可点击
- **输出结果：** 回复被发送或展示在推荐回复文本框中

**自动回复优先级：**

```
特定对象的自动回复设置 > 模式的全局自动回复设置 > 设置界面的全局自动回复设置
```

#### F004 多模式运行管理

- **触发条件：** 用户在模式子界面点击"启动"或"停止"按键
- **前置条件：** 模式处于"已启用"状态
- **主要流程：**
  1. 用户点击"启动"按键
  2. 系统创建该模式的 RuntimeHost 实例（多实例单进程架构）
  3. RuntimeHost 共享全局 DesktopDevice（加锁互斥）和 AIClient
  4. 模式进入"运行中"状态，参与红点检测和回复路由
  5. 用户点击"停止"按键
  6. 系统销毁该模式的 RuntimeHost 实例
  7. 模式进入"已停止"状态，不再参与红点检测和回复路由
- **异常处理：**
  - 设备被其他模式占用时 → 等待设备释放（串行处理）
  - 情感分析懒加载失败 → 降级为不使用情感分析，记录日志
- **输出结果：** 模式正确启动或停止

**两层启停机制：**

| 层级    | 操作            | 位置        | 效果                             |
| ----- | ------------- | --------- | ------------------------------ |
| 启用/禁用 | 切换 enabled 状态 | 设置界面-模式管理 | 控制是否在左侧菜单展示、是否创建子界面组件          |
| 启动/停止 | 切换 running 状态 | 模式子界面     | 控制是否创建 RuntimeHost 实例、是否参与回复路由 |

**并发模型：多实例单进程**

- 所有模式在主进程中运行，每个模式有独立的 RuntimeHost 实例
- 共享 DesktopDevice（加锁互斥，串行操作）
- 共享 AIClient（无状态，无需互斥）
- 共享 SentimentClassifier（懒加载单例，串行使用）
- VLM 布局检测只执行一次，所有模式共享 LayoutCache

**情感分析懒加载策略：**

- 引擎启动时不加载情感分析模型
- 首次有模式需要使用情感分析时，才启动 Python 子进程加载 BERT 模型
- 模型加载后保持运行，直到所有使用情感分析的模式都停止
- 多个模式串行复用同一个 SentimentClassifier 实例

## 3. 非功能需求

### 3.1 性能需求

- 模式切换响应时间：< 200ms（组件实例已创建，仅切换显示）
- 新增模式创建时间：< 500ms（仅数据持久化 + 组件实例化）
- 情感分析懒加载时间：< 35秒（首次，后续已加载则 < 1秒）
- VLM 对象名称识别延迟：< 5秒（在现有布局检测基础上增加一次 VLM 调用）
- 推荐回复生成延迟：与现有自动回复相同（取决于 Provider 和模型）

### 3.2 安全需求

- API Key 加密存储（electron-store 支持加密）
- 模式配置数据本地存储，不上传云端
- 特定对象名称仅在本地使用，不对外传输

### 3.3 可用性需求

- 系统预设模式不可删除，保证始终有可用模式
- 全局默认模式必须指向一个已启用的模式
- 模式运行异常时，不影响其他模式的运行（实例隔离）

## 6. 约束与风险

### 6.1 技术约束

- Electron 单进程架构限制：所有模式共享主进程，一个模式崩溃可能影响其他模式
- VLM 识别准确率限制：对象名称识别依赖 VLM 视觉能力，可能存在误识别
- 微信单窗口限制：同一时间只能操作一个聊天窗口，所有模式必须串行处理
- 剪贴板互斥：粘贴操作使用系统剪贴板，同一时间只能有一个粘贴操作

### 6.2 资源约束

- 情感分析模型内存占用：400-700MB（BERT 模型）
- VLM API 调用成本：每次识别消耗 API token
- 开发人力：单人开发，需控制功能范围

### 6.3 已识别风险

| 风险描述                       | 影响程度 | 应对策略                        |
| -------------------------- | ---- | --------------------------- |
| VLM 对象名称识别准确率不足            | 高    | 提供手动修正机制，识别结果与配置名称模糊匹配      |
| 多模式共享设备时死锁                 | 高    | 设备操作加锁超时机制，超时自动释放           |
| 情感分析懒加载延迟影响首次回复            | 中    | 异步加载，加载完成前降级为不使用情感分析        |
| 模式数量增多导致 UI 卡顿             | 中    | 未启用模式不创建组件实例，虚拟化列表          |
| electron-store 数据量增大导致读写变慢 | 低    | 模式和模型数据量有限（预计 < 100条），影响可忽略 |

## 8. 附录

### 8.1 术语表

| 术语     | 英文                            | 定义                                   |
| ------ | ----------------------------- | ------------------------------------ |
| 模式     | Reply Mode                    | 一组完整的会话运行配置，包含 prompt、情感分析开关、特定对象列表等 |
| 特定对象   | Specific Object               | 绑定到某个模式的聊天对象，系统识别后自动路由到对应模式          |
| 全局调度器  | Mode Scheduler                | 负责红点检测、对象识别、模式路由的全局组件                |
| 模式运行时  | Mode Runtime                  | 一个模式的 RuntimeHost 实例及其关联资源           |
| 推荐回复   | Recommended Reply             | AI 生成但未自动发送的回复文本，展示在推荐回复文本框中         |
| 一键粘贴   | One-Click Paste               | 将推荐回复自动粘贴到聊天输入框但不发送                  |
| 一键回复   | One-Click Reply               | 将推荐回复自动粘贴到聊天输入框并发送                   |
| 懒加载    | Lazy Loading                  | 延迟到首次使用时才初始化资源（如情感分析模型）              |
| 多实例单进程 | Multi-Instance Single Process | 多个 RuntimeHost 实例在同一进程中运行，共享设备等资源    |
| 启用/禁用  | Enable/Disable                | 控制模式是否在 UI 中展示和创建组件实例                |
| 启动/停止  | Start/Stop                    | 控制模式 RuntimeHost 是否运行，是否参与回复路由       |

| 序号 | 决策内容    | 选择                  | 理由                      |

| 8  | 模式子界面渲染 | 多实例切换               | 各模式状态独立保留               |

| 11 | 系统模式    | 预设不可删，可禁用           | 保证始终有可用模式               |
| 12 | 启停语义    | 两层分离（启用/禁用 + 启动/停止） | 展示控制与运行控制解耦             |
| 13 | 目标应用    | 全局统一配置              | 现有实现完善，无需修改             |
| 14 | 一键粘贴    | 自动粘贴不发送             | 用户可修改后再手动发送             |
| 15 | 统一开头    | 模式配置项，非必填           | 灵活标识自动回复消息              |
